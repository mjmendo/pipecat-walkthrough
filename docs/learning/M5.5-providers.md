# M5.5 Learning Guide — Provider Exploration (Swap & Measure)

## Setup: Keep Grafana open throughout

```bash
./scripts/start-infra.sh
python bots/tech-support/observability_server.py
# Grafana: http://localhost:3000 — TTFB histograms will update in real time
```

Each example is a 3-line swap in `observability_server.py`. Keep the other two services on OpenAI so you isolate one variable at a time.

---

## Example 1 — Swap STT: Deepgram Nova-3

**Do:** Replace `OpenAISTTService` with `DeepgramSTTService`:

```python
# Before:
from pipecat.services.openai.stt import OpenAISTTService
stt = OpenAISTTService(api_key=os.getenv("OPENAI_API_KEY"), model="gpt-4o-transcribe")

# After (3 lines):
from pipecat.services.deepgram.stt import DeepgramSTTService
stt = DeepgramSTTService(api_key=os.getenv("DEEPGRAM_API_KEY"))
# Add DEEPGRAM_API_KEY to .env (free tier: 12k min/yr at console.deepgram.com)
```

**Observe in Grafana:** STT TTFB histogram drops from ~420ms to ~180ms. The p50 line visibly shifts downward.

**Observe in stdout:**
```
Before: [METRICS] TTFB | processor=OpenAISTTService  value=0.42s
After:  [METRICS] TTFB | processor=DeepgramSTTService value=0.18s
```

**Understand:**
Deepgram Nova-3 is streaming-native: it starts transcribing before you finish speaking (streaming ASR). OpenAI Whisper is batch-then-respond: it waits for a speech segment to end, then sends the whole chunk to the model. Streaming = lower TTFB because the model is working in parallel with your speech.

The pipeline doesn't change — only the service at the STT slot. Pipecat's frame contract (`TranscriptionFrame`) is identical regardless of provider.

## Example 2 — Swap STT: Local Whisper.cpp

**Do:** Install and run Whisper.cpp, then swap:

```python
# Install: pip install pipecat-ai[whisper]
from pipecat.services.whisper.stt import WhisperSTTService
stt = WhisperSTTService(model="base.en")  # Downloads model on first run
```

**Observe in Grafana:** STT TTFB jumps to 800ms–2s depending on hardware. On Apple Silicon M-series, `small.en` runs at ~600ms. On CPU-only, `base.en` can exceed 2s.

**Observe:** Zero API calls. No data leaves your machine. No API key needed.

**Understand:**
Local inference trades latency for cost and privacy. The model must run on your hardware — if you're on Apple Silicon (M1/M2/M3), the Neural Engine accelerates it significantly. On a CPU-only machine, it's often too slow for real-time voice.

This is the key tradeoff table for STT:

| Provider | TTFB | Cost | Privacy | Notes |
|----------|------|------|---------|-------|
| Deepgram Nova-3 | ~180ms | Free tier | Cloud | Best TTFB |
| OpenAI gpt-4o-transcribe | ~420ms | Pay-per-use | Cloud | Good accuracy |
| Whisper.cpp (local) | 600ms–2s | Free | 100% local | Apple Silicon: faster |

## Example 3 — Swap LLM: Ollama (Llama 3.2)

**Do:** Install Ollama, pull a model, then swap:

```bash
# Terminal 1:
brew install ollama && ollama serve
ollama pull llama3.2  # 2GB download
```

```python
# In observability_server.py:
from pipecat.services.ollama.llm import OllamaLLMService
llm = OllamaLLMService(model="llama3.2")
# No API key needed — talks to localhost:11434
```

**Observe in Grafana:** LLM TTFB increases vs GPT-4o baseline. On Apple Silicon M-series, expect 800ms–1.5s for first token. On CPU-only, 3–5s+.

**Observe:** No OpenAI LLM calls in your API usage dashboard. Conversation history stays on-machine.

**Understand:**
Ollama runs the model locally. TTFB is determined by your hardware — specifically, how fast it can load the model into memory and generate the first token. Apple Silicon's unified memory architecture makes local inference practical for 7B-parameter models.

The function calling behavior may differ: smaller models sometimes fail to call `transfer_to_pizza()` reliably. If you see the pizza transfer not triggering, the model may not support function calling well. Try `llama3.2` (3B) vs `llama3.1:8b` to compare.

## Example 4 — Swap TTS: Cartesia Sonic

**Do:** Sign up for Cartesia free trial, then swap:

```python
# Install: pip install pipecat-ai[cartesia]
from pipecat.services.cartesia.tts import CartesiaTTSService
tts = CartesiaTTSService(
    api_key=os.getenv("CARTESIA_API_KEY"),
    voice_id="a0e99841-438c-4a64-b679-ae501e7d6091",  # British Lady
)
# Add CARTESIA_API_KEY to .env
```

**Observe in Grafana:** TTS TTFB drops from ~300ms to ~80ms. The p50 line drops sharply.

**Observe:** Voice quality and character are different. Test your persona prompts — "Hi, I'm Alex from tech support" may sound more or less natural depending on the voice ID.

**Understand:**
Cartesia Sonic generates audio at the phoneme level and streams it. The first audio chunk arrives before the sentence is fully synthesized. OpenAI TTS waits for a complete sentence chunk before sending audio.

This matters for perceived latency: the user hears the first word much sooner with Cartesia. The total audio duration is similar — only the TTFB changes.

TTS voice IDs are provider-specific. Switching TTS providers means selecting a new voice. Test with your persona's actual greeting to verify it sounds appropriate.

## Example 5 — Full Local Stack

**Do:** Combine local providers:

```python
# Whisper.cpp STT
from pipecat.services.whisper.stt import WhisperSTTService
stt = WhisperSTTService(model="base.en")

# Ollama LLM
from pipecat.services.ollama.llm import OllamaLLMService
llm = OllamaLLMService(model="llama3.2")

# Coqui TTS (requires separate server)
# Install: pip install pipecat-ai[coqui]
# Run: tts-server --model_name tts_models/en/ljspeech/tacotron2-DDC
from pipecat.services.coqui.tts import CoquiTTSService
tts = CoquiTTSService(base_url="http://localhost:5002")
```

**Observe in Grafana:** Full pipeline TTFB visible. Compare vs full-OpenAI baseline:

| Metric | Full OpenAI | Full Local (Apple M-series) |
|--------|-------------|------------------------------|
| STT TTFB | ~420ms | ~600ms |
| LLM TTFB | ~700ms | ~1200ms |
| TTS TTFB | ~300ms | ~400ms |
| Total perceived | ~1.4s | ~2.2s |

**Observe:** Zero API calls. All traffic on-machine. No network dependency.

**Understand:**
A fully local voice bot is viable on Apple Silicon M-series hardware. The latency is higher — typically 1.5–2.5x vs cloud — but acceptable for many use cases.

The tradeoff:
- **Cost:** Local = free after hardware. Cloud = pay-per-use.
- **Privacy:** Local = no data leaves machine. Cloud = audio/text to API provider.
- **Latency:** Local = higher TTFB, especially LLM. Cloud = lower TTFB with faster models.
- **Quality:** Local models are good but generally behind the frontier cloud models.

Now you have real numbers to back this table. The Grafana dashboard makes the difference visual — not just a benchmark claim, but your own measured data from your own voice bot.

## How to Read the Grafana TTFB Histogram

The `TTFB p50 by Service` panel shows the median TTFB for each service (STT, LLM, TTS) over the last 15 minutes. When you swap a provider, watch for:

1. **The p50 line dropping or rising** — this is the median user experience
2. **The p95 line** (second panel) — this is the worst-case. A provider with good p50 but bad p95 is unpredictable.
3. **Line stability** — a flat line means consistent latency. Jagged = high variance.

LLM TTFB is always the most variable. It grows with prompt length and output complexity. STT and TTS are more stable because they're bounded by audio duration and model inference time, which is roughly constant per token.
