# M6 — RTVI Protocol + Observers

## What RTVI Adds

The bot already works without RTVI. RTVI adds a **structured event layer** between the pipeline and the browser — so the UI knows exactly when the bot starts speaking, what it said, and when application events like call transfers happen.

Without RTVI: browser sees raw audio bytes.
With RTVI: browser receives typed events (`bot-started-speaking`, `user-transcription`, custom `transfer` events).

---

## Two RTVI Components

| Component | Type | Role |
|-----------|------|------|
| `RTVIProcessor` | `FrameProcessor` (in-path) | Handles incoming RTVI messages from client; manages handshake |
| `RTVIObserver` | `BaseObserver` (out-of-path) | Watches pipeline frames, converts them to RTVI events |

They work together: `RTVIObserver` is initialized with a reference to `RTVIProcessor`, which it uses to push RTVI messages to the transport output.

```python
rtvi = RTVIProcessor(transport=transport)

pipeline = Pipeline([
    transport.input(), stt, user_agg, llm, tts,
    rtvi,               # ← in the frame path
    transport.output(), assistant_agg,
])

task = PipelineTask(pipeline, observers=[
    RTVIObserver(rtvi),  # ← out-of-path observer
    DebugFrameObserver(),
])
```

## RTVI Events the Browser Receives

| Pipecat Frame | RTVI Event |
|---------------|------------|
| `BotStartedSpeakingFrame` | `bot-started-speaking` |
| `BotStoppedSpeakingFrame` | `bot-stopped-speaking` |
| `UserStartedSpeakingFrame` | `user-started-speaking` |
| `UserStoppedSpeakingFrame` | `user-stopped-speaking` |
| `TranscriptionFrame` | `user-transcription` |
| `LLMTextFrame` | `bot-llm-text` |
| `TTSTextFrame` | `bot-tts-text` |
| Custom (via `send_server_message`) | `server-message` |

All of these are generated by `RTVIObserver.on_push_frame()` — it watches the frame stream and emits RTVI JSON messages to the client for each relevant frame type.

## Custom Server Messages

To send application-level events to the browser (e.g., call transfer notification):

```python
await rtvi.send_server_message({
    "type": "transfer",
    "destination": "pizza",
    "message": "Transferring you to our pizza ordering service…",
})
```

On the browser side (pipecat-ai/client-js):
```javascript
client.on(RTVIEvent.ServerMessage, (data) => {
    if (data.type === "transfer") {
        showToast(data.message);  // "Transferring to pizza bot…"
    }
});
```

## DebugFrameObserver — Observer vs Processor Comparison

`DebugFrameObserver` is a learning tool in `bots/shared/debug_observer.py`. It logs every frame flowing through the pipeline with type, direction, and processor names.

**As an observer (zero latency):**
```python
task = PipelineTask(pipeline, observers=[DebugFrameObserver()])
```

**As a processor (adds inline latency):**
```python
class DebugProcessor(FrameProcessor):
    async def process_frame(self, frame, direction):
        logger.debug(f"[FRAME] {type(frame).__name__}")
        await self.push_frame(frame, direction)  # must re-push
```

The observer version runs in its own async task — the pipeline never waits for it. The processor version runs in the frame path and adds latency to every frame. For debugging, always use observers.

## RTVI Protocol Handshake

```
Browser → Server: { type: "client-ready" }
Server → Browser: { type: "bot-ready", version: "...", config: [...] }
--- pipeline runs ---
Server → Browser: { type: "bot-started-speaking" }
Server → Browser: { type: "bot-tts-text", data: { text: "Hi, how can..." } }
Server → Browser: { type: "bot-stopped-speaking" }
Browser → Server: { type: "action", action: "run_action", ... }
```

The handshake is managed by `RTVIProcessor.process_frame()` when it receives a `StartFrame`. Bot-ready is sent after `_send_bot_ready()` which is triggered by `on_bot_started` event handlers.
